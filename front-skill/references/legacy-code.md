# 旧代码处理与增量演进

## 默认原则

新增代码和本次修改触达代码必须遵守当前规范；不要为了追求统一大面积改动无关旧代码。

## 局部一致优先

- 优先遵循当前目录、当前模块既有写法。
- 在触达范围内做小范围收口。
- 禁止跨目录强推单一风格。
- 不因为发现旧代码不理想，就顺手重写大面积无关逻辑。

## 增量演进

改旧页面时优先收口到当前正式实现。旧结构不合适时，先指出问题，再做与当前任务相关的调整。

## 不做无关重写

不要为了理想形态重写无关代码。以下情况尤其要克制：

- 只是命名风格不同，但不影响本次需求。
- 旧代码结构大但本次只改小功能。
- 大范围替换会增加回归风险。
- 用户没有要求重构。

## 入口归一化

允许在边界入口做一次默认值归一化，主流程只消费规范化后的单一结构。

推荐：

```ts
const normalizedQuery = normalizeQuery(rawQuery)
```

主流程中不要散落多层兜底。

## 禁止猜测式兼容

禁止在业务主链路散落：

```ts
a || b || c
foo?.bar ?? foo?.baz ?? foo?.value
```

如果确实存在多版本字段兼容，必须明确来源、边界和退出条件，优先封装到入口归一化函数。

## 清理废品

提交前清理：

- 无用 import
- 未使用变量
- 明确调试残留
- 已失效历史注释
- 不再服务业务的兼容代码

保留仍表达业务含义的现有中文注释；失真注释直接修正或删除。

## 验证要求

涉及 Tailwind、主题、基础组件皮肤、多语言、API 协议、权限、登录链路的改动，至少执行相关类型检查、构建或最小必要回归。
