# 测试规范

## 默认原则

测试关注行为和结果，不测试实现细节。

## 适合补测试的场景

- 修改复杂工具函数。
- 修改 composable。
- 修改状态管理逻辑。
- 修改数据转换函数。
- 修复 bug 并需要防止回归。
- 修改复杂表单、权限、路由守卫、请求流程。

## Vitest

Vue / Vite 项目优先使用 Vitest，前提是项目已有或适合接入。

测试结构：

```text
foo.ts
foo.test.ts
```

## Vue 组件测试

- 测用户行为，不测内部变量名。
- 断言渲染结果、事件触发、状态变化。
- 异步更新后使用 `nextTick` 或 `flushPromises`。
- 不滥用 snapshot。

## Pinia 测试

- 测 store 的 state、getter、action 行为。
- 组件测试中需要正确初始化 Pinia。
- 不依赖测试执行顺序。

## E2E

复杂业务流程、登录、支付、关键表单、跨页面流程适合 E2E。是否使用 Playwright 或其他工具取决于项目已有配置。

## 不强行测试

小的样式调整、文案调整、无逻辑变更不强行补测试，但仍可运行 lint/build 验证。
