# 多语言规范

## 默认原则

用户可见文案必须优先走项目已有国际化体系。不要在页面、组件、表单 schema 或配置中私自硬编码多语言文案。

## 必须国际化的内容

新增或修改以下用户可见内容时，优先接入项目现有语言包或运行时翻译机制：

- 页面标题
- 按钮文案
- 提示语
- placeholder
- empty 文案
- toast / message / notification
- 表单校验提示
- 表格列标题
- 表单 schema 标签
- 弹窗标题和操作文案
- 状态文案和枚举展示值

## 语言来源统一

- 语言切换选项、语言列表、语言名称优先复用现有字典或共享常量。
- 禁止页面内私自写死一份 options。
- 配置型输入优先复用现有 `i18nCode` / `*I18nCode` 字段、现有多语言弹窗与输入组件。

## 运行时翻译

业务可配置文案、后端返回的多语字段、运行时翻译文案，遵循项目已有运行时翻译表和对应模块，不要并行发明新机制。

## 基础数据边界

只有新增或修改运行时翻译标记、语言字段、字典语言选项、多语配置时，才必须同步数据库基础数据、字典数据、初始化脚本或同步脚本。

## 新增语言种类

如果需求超出现有支持语言，必须同时评估：

- 共享类型
- 前端语言加载
- 后端翻译表结构和映射
- 语言字典初始化
- 运行时配置和默认语言策略

## 例外

开发调试日志、内部变量名、不可见配置键不需要国际化。业务上确认只展示给开发者的内容，可以保留原文，但要避免误进入用户界面。
