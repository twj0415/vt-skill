# 项目文档搭建

先判断：本次任务是“新建体系”还是“整理已有体系”。

- 如果项目还没有稳定文档结构，先新建最小体系。
- 如果项目已经有很多文档，先保留原文档，再重构入口、状态和交接层。
- 如果用户只要最简单可用方案，默认落最小体系，不主动扩成大而全规范库。
- 如果用户只给出一句模糊项目想法，先渐进确认，不直接开始设计或生成文档。

## 最小目标

先建立一套让后续 AI 不迷路的最小文档系统，而不是一次性写满全部设计稿。

最小可用集合固定为：

```text
doc/README.md
plan/README.md
plan/CURRENT.md
plan/HANDOFF.md
```

如果项目复杂度已经明显超过单一主线，再追加：

```text
decisions/
status/project-status.yaml
doc/modules/
doc/contracts/
```

不要在一开始同时创建过多目录。只有当项目复杂度、协作人数、技术栈分支或状态恢复成本已经上升时，才扩展结构。

## 执行流程

### 第一步：渐进确认关键信息

在设计文档体系前，先读取 `references/confirmation-gates.md`，按轮次确认关键信息。

确认规则：

1. 先判断用户输入的信息完整度；说清楚的内容直接采纳，不重复确认。
2. 提问前先做场景推断，例如扫描/拍照优先移动端，本地大文件优先桌面端，后台管理优先 Web。
3. 每轮最多问 3 个真正影响路线的问题；如果只缺 1 个关键点，就只问 1 个。
4. 需要提问时必须给出推荐和一行原因；存在合理分歧时，再给 1 到 3 个其他选项和一行原因。
5. 用户没有回答的问题标为“待确认”，不要猜成事实。
6. 用户明确说“你来定”时，才可以基于专业判断选择推荐方案，并说明取舍。
7. 必须先输出“项目启动确认单”或简短设计确认，等用户确认后，再进入详细设计。
8. 详细设计也需要二次确认，确认后才生成或修改 `doc/`、`plan/` 文件。

禁止行为：

- 不要因为用户说“帮我做一个记账 app”就直接写完整方案。
- 不要一次抛出十几个问题。
- 不要只列选项不表态。
- 不要写大段解释；每个推荐原因控制在一行。
- 不要为了格式强行提问；用户已经明确的内容直接采纳。
- 不要把 demo、小工具和正式产品按同一套工程标准设计。
- 不要在未确认组件库、技术栈、运行端、工程级别前固定目录和阶段计划。

### 第二步：判断项目类型

先判断项目属于哪一类：

- web
- uniapp
- desktop
- fullstack
- multi-app

如果同时存在多个端：

- 先定义一个主项目类型。
- 再定义每个端是“主应用”“管理端”“桌面端”“移动端”还是“配套工具”。
- 不要把多端差异直接揉进一篇总文档，改为主文档概览 + 分端补充。

涉及项目类型差异时，读取 `references/project-type-guidance.md`。

### 第三步：判断是新建还是整理

如果是新建项目：

1. 直接创建最小文档集合。
2. 先写目标、范围、当前主线、开发顺序。
3. 再写当前状态和交接机制。

如果是已有项目：

1. 先盘点现有 `doc/`、`plan/`、`README`、需求稿、原型稿、数据库设计、接口文档。
2. 找出唯一入口缺失、当前状态缺失、交接记录缺失、计划散落这四类问题。
3. 保留旧文档内容，不先删；先补入口和状态层。
4. 只有在新入口稳定后，才逐步迁移或归档旧文档。

### 第四步：固定单一入口

必须明确区分三类信息：

- `doc`：长期相对稳定的事实、目标、结构、规范。
- `CURRENT`：现在正在做什么。
- `HANDOFF`：上一轮做到哪、为什么停、下一轮先做什么。

如果一个项目没有 `CURRENT.md`，默认它不具备稳定续做能力。

### 第五步：写最小体系

按照 `references/minimal-doc-system.md` 的模板创建最小文件。

写的时候遵守：

1. `doc/README.md` 不写当天进度，只写长期信息。
2. `plan/README.md` 只写阶段顺序和执行纪律，不写会频繁变化的细项。
3. `plan/CURRENT.md` 只保留一个当前焦点，不允许同时挂多个“正在做”。
4. `plan/HANDOFF.md` 按时间倒序或顺序追加，但格式必须固定。

### 第六步：按复杂度决定是否扩展

当项目出现以下任一情况时，扩展结构：

- 有多个端或多个应用。
- 模块超过 8 个且强依赖明显。
- 已经存在复杂 DTO、数据库 schema、状态机、任务队列。
- 不同 AI 或不同窗口频繁接手。
- 用户明确要求“规定死”“换 AI 也能继续”。

这时增加：

- `decisions/`：记录关键架构决策，不把“为什么这样做”散落到各处。
- `status/project-status.yaml`：提供机器可读状态。
- `doc/modules/`：模块级规格。
- `doc/contracts/`：DTO、schema、枚举、状态、目录契约。

## 写作规则

所有文档都按下面的硬规则写：

1. 先写职责，再写内容，不写空泛前言。
2. 文件名直白，优先 `README`、`CURRENT`、`HANDOFF`、`ADR-xxx`、`模块名.md`。
3. 一个文件只承载一类职责，不混写“长期规则 + 当前状态”。
4. 需要后续持续更新的文件，必须有固定模板和固定字段顺序。
5. 不写“后续再说”“待定”这种无信息占位；如果未定，写清未定点和决策条件。
6. 不复制同一份规则到多个文件，单一事实来源必须明确。
7. 如果用户只要求最简单输出，不主动创建过多层级。

## 文档分层规则

### 适合放在 `doc/`

- 项目目标
- 用户问题
- 技术栈
- 模块划分
- 页面或功能总览
- 核心数据流
- 统一规范

### 适合放在 `plan/`

- 阶段顺序
- 当前进行中任务
- 阻塞项
- 下一步动作
- 交接记录

### 适合放在 `decisions/`

- 为什么选某个技术栈
- 为什么采用某种数据结构
- 为什么不采用替代方案

### 适合放在 `status/`

- 当前阶段
- 当前 TODO ID
- 已完成项
- 阻塞项
- 最后验证时间
- 下一动作

## 多项目和多端规则

如果项目是多端或多应用，不要直接为每个端复制整套文档。采用：

```text
doc/README.md
doc/apps/app-a.md
doc/apps/app-b.md
plan/CURRENT.md
plan/HANDOFF.md
```

统一规则：

1. 先写总目标和总边界。
2. 再写每个端的职责边界。
3. 当前开发焦点只允许有一个主线。
4. 如果多个端并行，必须在 `CURRENT.md` 明确“当前主线端”和“非当前端状态”。

## 交付要求

完成本 skill 任务时，至少交付：

1. 创建或重构后的目录方案。
2. 核心文件正文。
3. 对每个文件职责的简短说明。
4. 如果是已有项目，说明旧文档保留、迁移还是归档。

如果用户没有要求复杂版本，优先交付最小体系，不主动扩展为大型文档库。
