# 渐进确认门禁

本文件用于防止 AI 在需求模糊时直接开始设计。目标是少量多轮确认，而不是一次性问卷。

## 总原则

1. 每轮最多问 3 个问题。
2. 先确认会影响全局架构的问题。
3. 已知信息先复述，缺失信息再提问。
4. 用户回答后，更新确认单，再进入下一轮。
5. 每个问题必须给 1 个推荐选项和 2 到 3 个其他选项。
6. 每个选项必须有一行原因，不能只列名称。
7. 用户明确授权“你来定”时，才可以替用户选择默认方案。
8. 没确认的内容只能写为“待确认”，不能写成事实。

## 固定交互格式

每个问题必须使用下面的紧凑格式：

```text
1. 问题标题
推荐：选项 A —— 一行说明为什么推荐。
其他：选项 B —— 一行说明适合什么情况。
其他：选项 C —— 一行说明适合什么情况。
其他：选项 D —— 一行说明适合什么情况。（可选）
请确认：用推荐，还是改成其他选项？
```

输出限制：

- 不要写长段背景解释。
- 不要写超过 3 个问题。
- 不要让用户自己从一堆名词里猜。
- 不要给超过 4 个选项。
- 如果已有明显答案，先给推荐，再让用户确认。

## 第一轮：项目定位

当用户只给出一句项目想法时，先问这一轮。

必须确认：

- 项目类型：web、uniapp、桌面端、全栈、多端。
- 使用目标：自己用、内部工具、商业产品、演示 demo、开源项目。
- 工程级别：快速 demo、可维护小项目、正式产品、超级工程化。
- 交付范围：只要文档、要原型、要代码、要部署、要发布包。

示例问法：

```text
我先确认 3 个关键点，确认后再继续设计。

1. 项目类型
推荐：uniapp / App —— 需要扫码、拍照、移动端高频使用时更合适。
其他：web —— 适合后台管理、多人协作、数据看板。
其他：桌面端 —— 适合本地文件处理、离线能力、重型工具。
其他：fullstack —— 适合账号、云同步、后台规则和多端联动。
请确认：用推荐，还是改成其他选项？

2. 工程级别
推荐：可维护小项目 —— 适合认真长期使用，但不一开始做过度工程化。
其他：快速 demo —— 适合验证想法，少文档、少测试、快出效果。
其他：正式产品 —— 适合上线、陌生用户、权限、安全和稳定性要求。
其他：超级工程化 —— 适合多人长期协作、复杂业务、强测试和发布体系。
请确认：用推荐，还是改成其他选项？

3. 本次交付范围
推荐：文档 + 阶段计划 —— 需求还没定死时先降低返工风险。
其他：原型 + 文档 —— 适合先确认页面和交互。
其他：MVP 代码 + 文档 —— 适合需求已清楚并准备马上实现。
请确认：用推荐，还是改成其他选项？
```

## 第二轮：技术约束

第一轮确认后，再问技术相关问题。

必须确认：

- 前端框架。
- 后端或本地运行方案。
- 数据存储方案。
- UI 组件库和样式方案。
- 是否需要登录、权限、同步、支付、外部 API。

如果用户没有技术偏好，也必须给 1 个推荐方案和 2 到 3 个其他方案，每项写一行原因。

## 第三轮：功能范围

技术方向确认后，再拆范围。

必须确认：

- MVP 必做功能。
- 后续增强功能。
- 明确不做的功能。
- 用户角色和核心使用流程。
- 数据导入、导出、备份要求。

要求：

1. 不要把所有想法都塞进第一阶段。
2. 每个功能标记为 `MVP`、`后续`、`不做`、`待确认`。
3. 如果用户说“都要”，必须拆阶段，不能把所有功能放进第一阶段。

## 第四轮：阶段与验收

范围确认后，再输出阶段拆分。

每个阶段必须写清：

- 阶段目标。
- 输入是什么。
- 输出是什么。
- 要实现哪些功能。
- 不实现哪些功能。
- 验收标准。
- 验证方式。

阶段粒度建议：

```text
阶段 0：项目文档和工程约束
阶段 1：最小可用闭环
阶段 2：数据持久化和核心体验
阶段 3：高级功能和可靠性
阶段 4：发布、测试、运维或打包
```

根据项目级别调整：

- 快速 demo：阶段可以少，验证可以轻。
- 可维护小项目：必须有清晰模块和基本验证。
- 正式产品：必须补安全、测试、错误处理、发布策略。
- 超级工程化：必须补契约、状态机、ADR、CI、观测、回滚和迁移策略。

## 第五轮：设计确认

在生成正式文档前，必须输出设计确认单。

确认单至少包含：

- 项目定位。
- 技术栈。
- 组件库或 UI 方案。
- 工程级别。
- MVP 范围。
- 后续范围。
- 明确不做。
- 阶段计划。
- 文档目录。
- 当前仍待确认的问题。

用户确认后，才允许生成或修改文档文件。

## 停止条件

遇到以下情况必须停下来确认：

1. 项目类型不清。
2. 工程级别不清。
3. 技术栈和组件库不清，且用户没有授权 AI 决策。
4. MVP 范围不清。
5. 用户要求的范围明显超过目标工程级别。
6. 需求涉及真实支付、真实用户隐私、密钥、外部 API、发布上线。
7. 设计草案没有得到用户确认。

## 默认取舍

只有用户明确说“你来定”时，才使用默认取舍。

默认取舍规则：

- 个人 demo：优先简单、少依赖、少目录。
- 可维护小项目：优先主流技术栈、清晰模块、基础测试。
- 正式产品：优先稳定架构、权限、安全、错误处理、测试和发布。
- 超级工程化：优先契约、自动化、可观测、迁移、回滚和交接文档。

不要把最高工程标准默认套到所有项目上。
