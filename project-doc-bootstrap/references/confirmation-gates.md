# 自适应确认门禁

本文件用于让 AI 在项目启动前先思考、再确认、再设计。目标是交互顺畅、问题必要、推荐有依据，避免把确认流程做成固定问卷。

## 核心原则

1. 先判断用户信息完整度，再决定问不问。
2. 用户已经明确的信息直接采纳，不重复确认。
3. 能从场景合理推断的内容，先给推荐，再让用户改。
4. 只有影响架构、范围、成本、安全或交付节奏的问题才提问。
5. 每轮最多问 3 个问题；如果只缺 1 个关键点，就只问 1 个。
6. 提问时必须给推荐项和原因；存在合理分歧时，再给 1 到 3 个其他选项和一行原因。
7. 不为了凑格式增加选项；如果只有一个合理选择，直接说明按它处理，并给原因。
8. 没确认的内容只能写为“待确认”，不能写成事实。

## 信息完整度判断

先把用户输入分成三档。

### 模糊输入

例子：

```text
帮我做一个食品配料检测工具
```

处理：

- 先做场景推断。
- 再问 2 到 3 个关键问题。
- 每个问题给推荐和原因。

### 中等清楚

例子：

```text
我要做一个 uniapp 食品配料检测工具，用拍照 OCR，自己用，先做 demo
```

处理：

- 不再问项目类型、主入口、工程级别。
- 只问仍会影响设计的缺口，例如 OCR 方案、规则来源、是否保存历史。

### 很清楚

例子：

```text
我要做 uniapp + Vue3 + Pinia，本地 OCR，自己用，不要后端，只要文档和阶段计划
```

处理：

- 不进入第一轮问答。
- 直接输出“已采纳信息 + 少量风险确认 + 设计确认单”。
- 如果风险不高，可以直接进入文档设计草案。

## 先推断再提问

提问前必须先从需求中提取线索。

常见推断：

- 出现“扫描、拍照、相册、摄像头、条形码、二维码”：优先推荐手机端、uniapp 或 App。
- 出现“后台、管理、表格、多人协作、数据看板”：优先推荐 Web。
- 出现“本地文件、批量处理、离线、大文件、FFmpeg、桌面自动化”：优先推荐桌面端。
- 出现“账号、云同步、多设备、后台规则、多人数据”：优先推荐 fullstack。
- 出现“自己用、临时验证、先看看”：工程级别偏轻。
- 出现“上线、真实用户、支付、隐私、健康、安全、风险判断”：工程级别不能太低。
- 出现“插件、扩展、脚本、自动化”：先确认运行环境和分发方式。

推断后再问。不要把“Web / 桌面 / 手机 / 全栈”原样丢给用户。

## 推荐表达规则

推荐不是固定模板，但必须满足：

- 有明确推荐。
- 有一行推荐原因。
- 其他选项只在确实合理时给。
- 每个其他选项也要有一行适用原因。
- 不写长段解释。

可用写法：

```text
我先按场景判断：这个工具核心动作是拍照/扫描食品包装，所以主入口更适合手机端。

需要你确认项目类型：
推荐：uniapp / App。原因：拍照、相册、随手查询都在手机端更自然。
其他：fullstack。原因：如果需要账号、云端历史、后台规则库，就需要后端。
其他：Web。原因：更适合作为管理后台，不适合作为扫描主入口。

你确认用推荐方案，还是改成其他方向？
```

如果用户已经明确说“做 uniapp”，不要再问项目类型：

```text
项目类型我按 uniapp 采纳，不再重复确认。
还需要确认 OCR 和规则来源，因为它们会影响技术方案和阶段计划。
```

## 第一轮应该问什么

第一轮只问真正会影响整体路线的问题。

优先级：

1. 项目类型或运行端。
2. 工程级别。
3. 本次交付范围。

如果用户已经说清其中某项，就跳过，不要凑满 3 个问题。

## 第二轮应该问什么

第二轮聚焦技术方案，只问当前阶段需要决定的内容。

可能包括：

- 前端框架或应用框架。
- 后端、本地运行或云端方案。
- 数据存储。
- 组件库和样式方案。
- 核心第三方能力，例如 OCR、LLM、地图、支付、通知。

不要过早问细枝末节。比如只是要文档和阶段计划时，不需要立即追问所有表字段。

## 第三轮应该问什么

第三轮聚焦范围和阶段。

必须把功能拆成：

- MVP 必做。
- 后续增强。
- 明确不做。
- 待确认。

如果用户说“都要”，必须拆阶段，不能把所有功能塞进第一阶段。

## 设计确认

生成正式文档前，输出设计确认单。

确认单应该简洁但完整：

- 已采纳信息。
- AI 推断和推荐。
- 用户已确认项。
- 仍待确认项。
- MVP 范围。
- 后续范围。
- 阶段计划。
- 文档目录。

如果用户已经给得很清楚，设计确认单可以很短，不要强行展开成大文档。

## 停止条件

遇到以下情况必须停下来确认：

1. 项目类型或运行端无法判断，且会影响技术路线。
2. 工程级别不清，且会影响文档深度、测试、发布和安全设计。
3. 用户要求的范围明显超过目标工程级别。
4. 需求涉及真实支付、真实用户隐私、健康风险、法律风险、密钥、外部 API 或发布上线。
5. 技术栈和组件库不清，且用户没有授权 AI 决策。
6. MVP 范围不清。
7. 设计草案没有得到用户确认。

## 输出质量要求

每次交互都要做到：

- 先说明自己基于哪些线索做判断。
- 问题少，但问到会影响方向的点。
- 推荐有立场，原因简短。
- 不重复问用户已经说清楚的内容。
- 不因为模板牺牲交互体验。
- 不因为交互少而省略关键风险。
